================================================================================
                    1. SECURITY: LOCKING DOWN THE PRIVATE GYM
================================================================================
An athlete's body weight, metrics, medical context, and workout routines are highly personal. We must protect this data like a top-security VIP locker room.

* CURRENT STATE:
  - The application stores user weights, logs, and profile states in the browser's `localStorage` and sends email requests via clean API endpoints.

* STRENGTH COACH SUGGESTIONS for Bulletproof Security:
  - **HttpOnly JWT Session Tokens:** Shift session management away from raw `localStorage` (which is vulnerable to XSS cross-site scripting attacks). When a user logs in with their OTP, sign a secure JSON Web Token (JWT) on the server and set it as an `HttpOnly` and `Secure` cookie.
  - **Environment Variables Lock:** Ensure the Groq API key is strictly isolated. We did an awesome job wrapping the model queries inside server-side NextJS API routes (`/api/gemini/analyze-routine`). Never let any client-side components directly access external LLM keys!
  - **End-to-End API Validation:** Implement runtime schema checkers (like Zod or Yup) inside our API routes. When a user submits an analysis request, validate the data schema immediately:
    ```ts
    const metricsSchema = z.object({
      weight: z.number().min(30).max(250),
      protein: z.number().min(0).max(500),
    });
    ```
    This completely blocks malicious query injections and secures server resources.

================================================================================
               2. SPEED: ZERO-LAG QUERY LATENCY ON THE GYM FLOOR
================================================================================
When a bodybuilder is resting between heavy sets, they only have 60 to 90 seconds. If an API request takes 4 seconds to load their routine, it ruins their workout flow.

* CURRENT STATE:
  - The API requests fetch fresh AI Timetables and analyze routines by calling the Groq API endpoint directly.

* STRENGTH COACH SUGGESTIONS for Speed Boosts:
  - **Dynamic Client-Side Caching (SWR or React Query):** Integrate caching hooks like `useSWR` or `React Query` to fetch logged meals and sets. If the user flips between the Diet and Dashboard tabs, serve the cached state instantly instead of re-fetching from local files or database clusters.
  - **Edge Caching for AI Recommendations:** AI recommendations only need to update when a user logs a new weight or logs meals. Implement **NextJS Incremental Static Regeneration (ISR)** or Edge Middleware caching. When a user queries `/api/gemini/analyze-routine`, verify if the cached daily routine exists in a Redis store first before hitting the expensive Groq LLM API.
  - **Minimize Database Query Payloads:** When exporting data in `/reports`, compile the logs server-side. Don't fetch unnecessary columns (like whole historical arrays). Only pull specific columns needed for that report, dropping query sizes by 80%.

================================================================================
              3. DATA HANDLING: SCIENTIFIC AND FAULT-TOLERANT DB
================================================================================
If a user switches their phone or clears their browser cache, they should NEVER lose years of tracking progress.

* CURRENT STATE:
  - The app depends entirely on local storage databases, making progress vulnerable to browser clears.

* STRENGTH COACH SUGGESTIONS for Dynamic Database Integration:
  - **Database Migration Plan (Prisma + Supabase):** Setup a PostgreSQL database hosted on Supabase and manage schemas with Prisma ORM. It’s lightweight, fast, and secure.
  - **Create a Clean Database Schema:**
    * `User`: Email, PasswordHash, Age, Height, TargetWeight, ActivityLevel.
    * `Plan`: Name, StartWeight, TargetWeight, Duration, Active (Boolean).
    * `DailyReport`: Date, CalorieScore, SleepScore, ProteinLogged, WaterLogged, FatLogged, OverallScore.
    * `WorkoutLog`: ExerciseName, WeightLogged, RepsCompleted, Timestamp.
  - **Offline-First Synchronization (PWA Capability):** Enable offline syncing. When the user logs an exercise in a basement gym with zero internet signal, save the set to an `IndexedDB` storage in the browser, and background-sync it to the cloud database the moment they walk outside and hit cellular data!
